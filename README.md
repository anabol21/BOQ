# BOQ — текущий статус проекта

## О чём проект
BOQ — это исследовательский проект по оптимизации кинорасписания, который связывает прогноз спроса с оптимизацией расписания. Текущий пайплайн проходит путь от проверки синтетических данных к ML-прогнозу, затем к классическому бейзлайну, а дальше — к QUBO / Ising и отжиг-экспериментам.

## Как устроен пайплайн
- `00_data_reality_and_validity_check.ipynb` проверяет, достаточно ли реалистичен и согласован synthetic raw dataset.
- `01_data_loading_and_split.ipynb` подготавливает типизированные и разбитые по времени датасеты для ML.
- `02_baseline_ml_model.ipynb` обучает базовую модель прогноза спроса / заполняемости.
- `03_classical_solvers_baseline.ipynb` строит классический бейзлайн для расписания и экспортирует QUBO-артефакты.
- `04_ising_formulation_and_annealing.ipynb` переводит QUBO в Ising и запускает отжиго-вдохновленные солверы.

## Текущий статус
На текущий момент в проекте уже собран сквозной исследовательский каркас: данные валидируются, бейзлайн ML-модель обучается, есть классический бейзлайн на CP-SAT, а также построена QUBO / Ising-постановка для отжиг-экспериментов.

Команда уже выявила важный практический вывод: физические пересечения в одном и том же зале должны трактоваться как жёсткие ограничения, иначе сравнение между классикой и отжиг-результатами становится некорректным.

Сейчас проект находится на стадии честного сравнения подходов. То есть основная задача — не просто получить «красивую энергию», а выстроить корректную, воспроизводимую и интерпретируемую логику сравнения солверов.

- Подбор и сравнение отжиговых солверов.
- Калибровка штрафов для софт ограничений и хард ограничений.
- Качество OpenJij / симуляции отжига относительно CP-SAT бейзлайна.
- Переход от отжиг-экспериментов к QAOA.

## На какие результаты смотреть в первую очередь
При чтении ноутбуков важно смотреть не только на метрику солвера, но и на декодированное качество расписания. Ключевые метрики проекта:
- суммарная выручка;
- число выбранных сеансов;
- число физических hall-overlap конфликтов;
- число same-movie simultaneous cross-hall конфликтов;
- согласованность между классикой и отжиг-постановками.

Низкая энергия важна только в том случае, если после декодирования решение остаётся физически реализуемым и бизнес-осмысленным.

## Как команде лучше знакомиться с проектом
1. Начать с этого README, чтобы понять структуру проекта и текущий статус.
2. Открыть notebook 00, чтобы понять качество данных.
3. Перейти к notebook 02, чтобы понять, откуда берётся прогноз выручки.
4. Затем открыть notebook 03 как главный бейзлайн.
5. После этого переходить к notebook 04 и анализировать отжиг-эксперименты.
